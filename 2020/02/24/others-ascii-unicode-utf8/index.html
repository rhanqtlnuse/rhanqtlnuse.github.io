ASCII、Unicode 和 UTF

发表于 2020-02-24 更新于 2020-02-28
如题

1    ASCII 码

ASCII 码是比较早（不是最早）的一种字符编码，最初占 8 位（计算机中最小的寻址单元）只支持 128 个字符（字母只支持英文字母），最高位被规定为 0

在计算机的上古时代，内存非常宝贵。为了节省内存，人们使用 BCDIC，也叫“6 位字符编码系统”。这套编码系统最多表示 26 = 64 种字符，减去数字，控制字符等等，发现剩下的不够52个英文字母，于是BCDIC干脆只保存大写字母。于是，一些这个时期诞生的语言：Assembly(汇编)、Fortran，都是整篇的大写。后来ASCII，也就是7位编码方案诞生了，人们有小写字母用了，回头一看以前的汇编代码全是大写，突然开始大小写混用还有点不习惯…一不做二不休，既然大写了就全大写，新人如果爱拿小写写，编译器就一律无视。编译的时候大小写干脆没区别。

1.2    扩展 ASCII 码

使用 ASCII 码会导致一个问题：除英文字母之外的文字无法表示，比如法语中带注音符号的字母和德语中的变音字母。所以，后来出现了「扩展 ASCII 码」，即在原来 ASCII 码的基础上，利用了剩余的 128 个（0x80 到 0xff）位置，编入了新的符号。比如，法语中的 é 的编码为 130（二进制 10000010）。这样一来，这些欧洲国家使用的编码体系，可以表示最多 256 个符号。

但是问题就这样出现了 —— 不同的国家有不同的字母，即使它们都使用 256 个符号的编码方式，代表的字母却不一样。比如，130 在法语编码中代表了 é，在希伯来语编码中却代表了字母 Gimel（ג），在俄语编码中又会代表另一个符号。

然而，在亚洲有更让人头疼的问题。亚洲语言有更多的字符和字形需要被存储 —— 仅汉字就多达 10 万左右，一个字节已经不够用了。所以他们开始使用两个字节来存储字符，这被称作 DBCS（双字节编码方案），比如，简体中文常见的编码方式是 GB2312（对 ASCII 的中文扩展）。在 DBCS 中，字符串操作变得很蛋疼 —— 应该怎么做 str++ 或 str—？

这些问题成为了系统开发者的噩梦。例如，为了把软件卖到全世界，MS DOS 必须支持所有风格的 ASCII 码。他们提出了「内码表」这一概念。例如，你需要告诉 DOS（通过 chcp 命令）你想使用保加利亚语的内码表，它才能显示保加利亚字母。内码表的更换会应用到整个系统。这对使用多种语言工作的人来说是一个问题，因为他们必须频繁的在几个内码表之间来回切换。

内码表并不是一个简洁的解决方案，它只是为了让编码系统可以正常工作而提出的一项 hack 技术或者说是简单的修正。

2    Unicode

Context

Key Concepts

Let’s level set on some ideas:

Ideas and data are different. The concept of “A” is something different than marks on paper, the sound “aaay” or the number 65 stored inside a computer.

One idea has many possible encodings. An encoding is just a method to transform an idea (like the letter “A”) into raw data (bits and bytes). The idea of “A” can be encoded many different ways. Encodings differ in efficiency and compatibility.

Know thy encoding. When reading data, you must know the encoding used in order to interpret it properly. This is a simple but important concept. If you see the number 65 in binary, what does it really mean? “A” in ASCII? Your age? Your IQ? Unless there is some context, you’d never know. Imagine if someone came up to you and said “65”. You’d have no idea what they were talking about. Now imagine they came up and said “The following number is an ASCII character: 65”. Weird, yes, but see how much clearer it is?

Embrace the philosophy that a concept and the data that stores it are different. Let it rustle around in your mind…

2.1    字符集 vs. 字符编码

Character set is an idea while encoding is data

在 ASCII 及之前的时代并不需要显式区分这两个概念

2.2    Unicode

为了解决各地（国）编码不统一的问题，ISO 提出了 Universal Multiple-Octet Coded Character set，简称 UCS，俗称 Unicode。

Unicode 就像一个电话本，标记着字符和数（官方术语是码位 (Code Point)）之间的映射关系。总是用U+开头。理论上每种语言中的每种字符都被Unicode协会指定了一个神奇数字。这种编码总是以 U+ 开头，比如「中」的 Unicode 编码就是 U+4e2d。

Unicode并不涉及字符是怎么在字节中表示的，它仅仅指定了字符对应的数字，仅此而已。

Unicode 规定使用 2 字节表示一个字符，所以对于 ASCII 中的半角字符，Unicode 维持其编码，只是将高字节设为全 0，这就引出了一个问题：存储空间的浪费。如果一个文本中全是 ASCII 字符，那么占用的存储空间将会有一半是浪费的！为了解决这个问题，就有了 UTF

关于Unicode的其它误解包括：Unicode支持的字符上限是65536个，Unicode字符必须占两个字节。告诉你这些的人应该去换换脑子了。严格来说，Unicode 是没有“长度”这一说的，它是抽象的字符，只有Unicode的编码才有具体的字节长度。

记住，Unicode只是一个用来映射字符和数字的标准。它对支持字符的数量没有限制，也不要求字符必须占两个、三个或者其它任意数量的字节。

Unicode字符是怎样被编码成内存中的字节这是另外的话题，它是被UTF(Unicode Transformation Formats)定义的。

Unicode itself is a mapping, it defines codepoints and a codepoint is a number, associated with usually a character. I say usually because there are concepts like combining characters. You may be familiar with things like accents, or umlauts. Those can be used with another character, such as an a or a u to create a new logical character. A character therefore can consist of 1 or more codepoints.

To be useful in computing systems we need to choose a representation for this information. Those are the various unicode encodings, such as utf-8, utf-16le, utf-32 etc. They are distinguished largely by the size of of their codeunits. UTF-32 is the simplest encoding, it has a codeunit that is 32bits, which means an individual codepoint fits comfortably into a codeunit. The other encodings will have situations where a codepoint will need multiple codeunits, or that particular codepoint can’t be represented in the encoding at all (this is a problem for instance with UCS-2).

Because of the flexibility of combining characters, even within a given encoding the number of bytes per character can vary depending on the character and the normalization form. This is a protocol for dealing with characters which have more than one representation (you can say "an 'a' with an accent" which is 2 codepoints, one of which is a combining char or "accented 'a'" which is one codepoint).

3    UTF

3.1    UTF-16 / UCS-2

UCS-2 / UTF-16 is nice and simple, but boy it does waste some bits. Not only does it double ASCII, but the converted ASCII might not even be readable due to the null characters.

另一个流行的可变长度编码方案是UTF-16，它使用2个或者4个字节来存储字符。然而，人们逐渐意识到UTF-16可能会浪费存储空间，但那是另一个话题了。

在UTF-8中，0-127号的字符用1个字节来表示，使用和US-ASCII相同的编码。这意味着1980年代写的文档用UTF-8打开一点问题都没有。只有128号及以上的字符才用2个，3个或者4个字节来表示。因此，UTF-8被称作可变长度编码。

3.2    UTF-8

The ‘8’ means it uses 8-bit blocks to represent a character.

UTF-8 remains a simple, single-byte, ASCII-compatible encoding method, as long as no characters greater than 127 are directly present. This means that an HTML document technically declared to be encoded as UTF-8 can remain a normal single-byte ASCII file. The document can remain so even though it may contain Unicode characters above 127, as long as all characters above 127 are referred to indirectly by ampersand entities.（所谓“ampersand entity”就是以 & 开头的特殊形式，比如“中”在这种表示法中就是 &#20013;）

Unicode 在很长一段时间内无法推广，直到互联网的出现，为解决 Unicode 如何在网络上传输的问题，于是面向传输的众多 UTF（UCS Transfer Format）标准出现了，顾名思义，UTF-8 就是每次 8 个位传输数据，而 UTF-16 就是每次 16 个位。UTF-8 就是在互联网上使用最广的一种 Unicode 的实现方式，这是为传输而设计的编码，并使编码无国界，这样就可以显示全世界上所有文化的字符了。UTF-8 最大的一个特点，就是它是一种变长的编码方式。它可以使用 1 到 4 个字节表示一个符号，根据不同的符号而变化字节长度，当字符在 ASCII 码的范围时，就用一个字节表示，保留了ASCII 字符一个字节的编码作为它的一部分（需要注意的是 Unicode 一个中文字符占 2 个字节，而 UTF-8 一个中文字符占 3 个字节）。从 Unicode 到 UTF-8 并不是直接的对应，而是要过一些算法和规则来转换：

Unicode（十六进制） UTF-8（二进制）
0000 0000 - 0000 007F 0xxxxxxx
0000 0080 - 0000 07FF 110xxxxx 10xxxxxx
0000 0800 - 0000 FFFF 1110xxxx 10xxxxxx 10xxxxxx
0001 0000 - 0010 FFFF 11110xxx 10xxxxxx 10xxxxxx 10xxxxxx

In any case, UTF-8 still needs a header to indicate how the text was encoded. Otherwise, it could be interpreted as straight ASCII with some codepage to handle values above 127. It still uses the U+FEFF codepoint as a BOM, but the BOM itself is encoded in UTF-8 (clever, eh?).

需要注意的是，Unicode 是字符集，而 UTF-8 是编码规则。字符集为每一个字符分配一个唯一的 ID（学名为码位/码点/code point），编码规则是将 code point 转换为字节序列的规则。

3.2    字符的存储

通常，在内存中使用 Unicode 来表示字符，而使用 UTF 来存储字符，这样：

  • 取字符时只需要总是取 2 个字节即可，节省了处理（计算）的时间，提高处理效率
  • 存储/传输字符时可以节省空间/带宽，提高传输效率

其中，Unicode 称为程序 “内码”，UTF 称为程序 “外码”。

小端 (Little Endian) vs. 大端 (Big Endian)

Endian读作End-ian或者Indian。这个术语的起源可以追溯到格列佛游记。（小说中，小人国为水煮蛋应该从大的一端（Big-End）剥开还是小的一端（Little-End）剥开而争论，争论的双方分别被称为“大端派”和“小端派”**。）

低字节序和高字节序只是一个关于在内存中存储和读取一段字节（被称作words）的约定。这意味着当你让计算机用UTF-16把字母A（占两个字节）存在内存中时，使用哪种字节序方案决定了你把第一个字节放在第二个字节的前面还是后面。这么说有点不太容易懂，让我们来看一个例子：当你使用UTF-16存下来自你朋友的附件时，在不同的系统中它的后半部分可能是这样的：

00 68 00 65 00 6C 00 6C 00 6F（高字节序，高位字节被存在前面）

68 00 65 00 6C 00 6C 00 6F 00（低字节序，低位字节被存在前面）

字节序方案只是一个微处理器架构设计者的偏好问题，例如，Intel使用低字节序，Motorola使用高字节序。

字节顺序标记(BOM)

如果你经常要在高低字节序的系统间转换文档，并且希望区分字节序，还有一种奇怪的约定，被称作BOM。BOM是一个设计得很巧妙的字符，用来放在文档的开头告诉阅读器该文档的字节序。在UTF-16中，它是通过在第一个字节放置FE FF来实现的。在不同字节序的文档中，它会被显示成FF FE或者FE FF，清楚的把这篇文档的字节序告诉了解释器。

BOM尽管很有用，但并不是很简洁，因为还有一个类似的概念，称作「魔术字」(Magic Byte)，很多年来一直被用来表明文件的格式。BOM和魔术字间的关系一直没有被清楚的定义过，因此有的解释器会搞混它们。

If you open a UCS-2 string and see FEFF, the data is in the right byte order and can be used directly. If you see FFFE, the data came from another type of machine, and needs to be converted to your architecture.

The BOM is actually a valid Unicode character – what if someone sent a file without a header, and that character was actually part of the file?

  1. 这个世界上从来没有纯文本这回事，如果你想读出一个字符串，你必须知道它的编码。
  2. Unicode 是一个简单的标准，用来把字符映射到数字上。Unicode协会的人会帮你处理所有幕后的问题，包括为新字符指定编码。
  3. Unicode并不告诉你字符是怎么编码成字节的。这是被编码方案决定的，通过UTF来指定。
  4. 还有最重要的：永远记得通过 Content-Type 或者 meta charset 标签来显式指定你的文档的编码。这样浏览器就不需要猜测你使用的编码了，他们会准确的使用你指定的编码来渲染文档。
