【译】揭开容器的神秘面纱 | 第一部分：内核空间

发表于 2020-03-29 更新于 2020-06-18
翻译自 Demystifying Containers - Part I: Kernel Space

原文地址

本系列博文和对应的讲座旨在从历史的角度给你提供一个关于容器的实用的视角。我们一起会逐层地（layer by layer）发现新型的云架构：从 Linux 内核的级别开始，直至编写我们自己的安全云原生（cloud native）应用。

结合历史背景的简单示例将指导你从最基本的 Linux 环境开始，直到制作安全的容器，这些容器非常适合当今和未来的编排（orchestration）世界。最后，理解 Linux 内核、容器工具、运行时、通过软件定义的网络和编排软件（比如，Kubernetes）中的特性是如何设计的以及这些特性是如何运转的会容易地多。

第一篇博文（和讲座）会讨论 Linux 内核相关的话题，这些话题会给你提供深入理解容器的必要的基础。我们会了解 UNIX、Linux 的历史，并讨论 chroot、名称空间（namespace）和 cgroup 之类的解决方案，并结合我们自己的示例。除此之外，我们将分析（peel）一些容器以了解我们以后要谈论的主题。

1    引言

在今天，说到容器，大多数人都会想到下面的大蓝鲸或是蓝色背景的舵轮。

让我们先把这些想法放在一边，问问自己：到底什么是容器？如果读读 Kubernetes 的文档，我们只能找到关于为什么要使用容器的解释和许多对 Docker 的引用。Docker 本身将容器解释为“标准的软件单元”（a standard unit of software）。它们的解释仅仅是提供了总体概述，并没能揭示底层的“魔法”。最终，人们会倾向于把容器当作廉价的虚拟机（virtual machine），但这种想象并不符合实际情况。无法精确地定义什么是容器不是完全没有道理的 —— 因为“容器”一词根本没有任何含义。容器编排生态系统中的“pod”一词也是如此。

如果我们简化一下，将容器看作只是在单个主机上运行的相互隔离的进程组，它们满足一些“共同”特性。其中一些神奇的功能直接内置于 Linux 内核中，并且绝大多数都具有不同的历史渊源。

因此，容器必须满足四个主要需求才能被接受：

  • 无可争议的是：容器必须在单个主机上运行。好的，因此两台计算机不能运行单个容器。
  • 显然的是：容器是进程组。你也许已经知道 Linux 进程存在于树结构中，因此我们可以说容器必须具有根进程（root process）。
  • 还好：容器应该是相互隔离的（isolated），不管“隔离”这个词到底是什么意思。
  • 不太清楚：它们必须具有共同的特性。一般而言，特性似乎会随着时间而变化，因此我们必须指出最常见的功能是什么。

单独列出这些要求会让人感到困惑，而且上面的图片也不够清楚。因此，让我们从历史上容器的起点着手，以保持简单。

2    chroot

基本上每个 UNIX 操作系统都可以更改当前正在运行的进程（及其子进程）的根目录。chroot 首先出现在 UNIX 版本 7（1979 年发行）中，后来又出现在 Berkeley 软件发行版（BSD）中。如今在 Linux 中，你可以以系统调用（内核 API 函数调用）或相应的独立包装程序的方式调用 chroot(2)。chroot 也被称为“监狱”（jail），因为在 1991 年有人将它用作监视安全骇客的蜜罐。因此 chroot 比 Linux 古老得多，并且在 21 世纪初被（误）用了第一种方法，运行我们今天称为“微服务”的应用程序。chroot 当前被各种应用程序使用，例如，在构建服务中用于不同的发行版。如今，BSD 的实现与 Linux 的实现有很大不同，我们将重点关注后者。

3    Linux 命名空间（namespace）

命名空间是 Linux 内核于 2002 年在 2.4.19 版本中引入一种功能。命名空间背后的想法是将某些全局系统资源包装在抽象层中。这使得命名空间中的进程看起来像具有自己的隔离资源实例。内核名称空间抽象允许不同的进程组具有不同的系统视图。

并非从一开始就实现了所有可用的命名空间。对如今我们所理解的“容器就绪”的完整支持于 2013 年在内核版本 3.8 中随着 user 命名空间的引入而完成。我们最终有七个不同的命名空间：mnt pid net ipc uts user 和 cgroup。不用担心，我们将详细讨论它们。在 2016 年 9 月，提出了另外两个尚未完全实现的名称空间（time 和 syslog）。在深入研究某些名称空间之前，让我们先看一下命名空间 API。

4    API

Linux 内核的名称空间 API 包含三个主要的系统调用：

4.1    clone

clone(2) 函数以类似于 fork(2) 的方式创建一个新的子进程。不同之处在于，clone(2) API 允许子进程与调用进程共享其执行上下文的一部分，例如内存空间、文件描述符表和信号处理程序表。你可以将不同的命名空间标志传给 clone(2) 来为子进程创建新的名称空间。

4.2    unshare

函数 unshare(2) 允许进程取消执行上下文中当前与其他进程共享的部分的关联。

4.3    setns

函数 setns(2) 将调用线程与提供的名称空间文件描述符重新关联。此函数可用于加入现有的名称空间。

4.4    proc 文件系统

除了可用的系统调用之外，proc 文件系统中包括了其他与名称空间相关的文件。从 Linux 3.8 开始，/proc/$PID/ns 中的每个文件都是一个“魔法”链接，可以用作对所引用的名称空间执行操作（如 setns(2)）的句柄（handle）。

这使我们能够进行诸如跟踪某些进程驻留在哪个名称空间中的操作。除以编程方式外，另一种使用名称空间的方法是使用 util-linux 软件包中的工具。这个软件包包含用于上述系统调用的专用包装程序。该程序包中一个与名称空间相关的便捷工具是 lsns。它能够列出当前所有可访问的名称空间或单个给定名称空间的有用信息。但是现在，让我们实践一下（get our hands dirty）。

5.3    进程间通信（ipc）

IPC 命名空间也是在 Linux 2.6.19 版本中引入的，并且隔离了进程间通信资源。特别的，是 System V IPC 对象和 POSIX 消息队列。这个命名空间的一个用例是分离两个进程之间的共享内存以避免误用。相反，每个进程都能够用在共享的内存段上使用相同的标识符，并产生两个不同的区域。当 IPC 命名空间被销毁时，命名空间中所有的 IPC 对象也都会被销毁。

5.4    进程 ID（pid）

PID 命名空间在 Linux 2.6.24（2008）版本中引入，给进程一个独立的进程标识符集合。这意味着不同命名空间中的进程可以有相同的 PID。最后每个进程有两个 PID：命名空间内部的 PID 和命名空间外、在主机系统上的 PID。PID 命名空间允许嵌套，因此如果一个新的进程被创建，它会在从当前的命名空间到最初的 PID 命名空间的每一个都有一个 PID

PID 命名空间中的第一个进程得到 1 号，并且会得到 init 进程的所有特殊待遇。比如，命名空间中的所有进程都会被交给本命名空间中的 PID 为 1 的进程号“领养”，而不是主机系统中的 PID 为 1 的进程。另外，某个 PID 命名空间中的 1 号进程若终止，该命名空间中的所有进程和后代都会终止。我们来创建一个新的 PID 命名空间。

每个网络接口（物理的或虚拟的）只会在每个命名空间中出现一次。在命名空间之间移动一个接口是可能的。每个命名空间都包含私有的 IP 地址集合、路由表、socket 列表、连接跟踪表、防火墙和其他网络相关的资源。

销毁网络命名空间时，会销毁所有的虚拟网络接口并且将所有的物理接口移动回最初的网络命名空间。

最后，user 命名空间为容器世界带来了极大的安全性提升，这对于运行无根容器至关重要。
