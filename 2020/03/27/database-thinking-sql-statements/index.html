1    SQL 的本质

1.1    SQL 语言、数据库和关系模型

  • SQL 是一种语言
  • 数据库的功能主要是存储数据，这些数据符合对现实世界的一部分所建立的特定模型。相应地，数据库必须提供可靠的基础设施，无论何时都能够让多个用户使用同一些数据，且在数据被修改时不破坏数据完整性。要求数据库能够处理来自不同用户的资源争用，并能在事务处理过程中遇到机器故障等极端情况下也保持数据一致性
  • 关系理论只关心如何根据查询条件取得正确的数据集，关系操作只负责准确无误地找出属于最终数据集的记录，关系理论所研究的是集合，关系理论并不涉及各种统计功能。关系操作适用于理论上无限大的、数学意义上的集合，而当我们使用排序、分组等操作时，这个结果数据集必须是一个有限数据集，已经不再是数学意义上的关系了

总结：千万别把 SQL 查询的执行过程中真正的关系操作和附加的展示层（presetation layer) 功能混为一谈

1.2    SQL 与优化器

以关系理论为支柱的优化器主要用于关系操作层

SQL 应该用来表达“要做什么”而非“如何来做”

关系理论的数学基础为数据处理提供了非常严谨的逻辑支持，因此 SQL 艺术本应注重减小“非关系操作层”的厚度，即尽量在关系操作层完成大部分处理，否则优化器在“非关系操作层”难以保证返回的结果数据和原始查询执行的结果一样。

在执行非关系操作时，应专注于那些解决问题所必需的数据，不要画蛇添足（因为有限数据集必须以某种方式进行临时存储，当内存所剩无几的时候，会引发硬盘数据交换等开销很高的活动，而且，索引所指的是磁盘地址，并非临时存储地址，所以数据一旦进行临时存储，就意味着我们向最快的数据访问方式说拜拜了（哈希索引可能例外））

1.3    优化器的有效范围

  • 优化器需要借助在数据库中找到的信息，包括普通统计数据和数据定义中重要的声明性信息。如果错误地将反映数据关系的重要语义信息写在触发器甚至应用代码中，优化器就无法利用这些重要信息
  • 能够进行数学意义上的等价变换，优化效果才能最佳
  • 优化器考虑整体响应时间（优化处理 + 查询执行），因此对于一些较为复杂的查询，优化器做出的可能并不是最优的选择
  • 优化器改善的是独立的查询

总结：如果是若干个小查询，优化器将逐个优化；如果是一个大查询，优化器会将它作为一个整体优化

1.4    掌握 SQL 艺术的五大要素

  • 数据总量：要考虑的最重要因素

  • 定义结果集的查询条件：为了定义结果集，必须从几个方面来考虑，包括过滤、主要 SQL 语句，以及庞大的数据量对查询的影响

  • 结果集的大小：也可以是 SQL 语句改动的数据量。无论何时，只要查询有可能返回空集时，都应该首先检查那个最大可能导致空结果集的条件，尤其是在该检查执行非常快捷时

    • 总结：熟练的开发者应该努力使响应时间与返回记录数成比例，这符合用户的期望

  • 表的数量：通常来说，现代的 DBMS 都能非常高效地连接很多表，但是在连接大量表时，会产生一些额外的问题

    • 需要连接多个表时，应该质疑设计的正确性

    • 对于优化器来说，随着表数量的增加，复杂度将指数增长（1.3 节）

    • 设计许多表的复杂查询时，如果可以用多种不同的方式进行连接，选择失误的可能性很高

    • 随着查询复杂度的增加，应该把查询拆成一系列独立视图。每个视图代表复杂查询的一部分

    • 当视图返回的数据远多于上级查询所需时，放弃使用该视图或者换用一个较简单的视图，通常可使效率大为改善

    • 总结：当视图返回不必要的元素时，别把视图内嵌在查询中，而是应将视图分解，将其组成部分加到查询主体中

  • 并发用户数

    • 考虑整体时间

2    过滤

2.1    过滤条件的含义

连接条件放在何处（WHERE 或 JOIN）应该都一样，然而其运行效率却会因优化器不同而异

条件计算顺序有时会很重要（比如依赖于前面条件的条件）

2.2    过滤条件的好坏

编写 SQL 语句时，应首先考虑的问题是：

  • 哪些数据是最终需要的，这些数据来自哪些表？
  • 哪些输入会传递到 DBMS 引擎？
  • 哪些过滤条件能滤掉不想要的记录？

对过滤条件进行排序，把能够尽快减少必须处理的数据量的条件放在前面

蝙蝠车买主

customers(customer_id, customer_name, city)

orders(order_id, customer_id, order_date)

articles(article_id, article_name, price)

order_detail(order_id, article_id, quantity)

问题：找出居住在 Gotham 市，并在最近 6 个月订购了蝙蝠车（article_name 为 Batmobile）的所有客户

  4. 由于使用了 DISTINCT，最后经过层层筛选的客户名还必须进行排序，以剔除重复项

注意，没有普适的解决方案 —— 甲之蜜糖，乙之砒霜 —— 表连接的方式取决于具体情况，决定因素是待处理数据的特点

（The Art of SQL 的作者认为）避免在最高层使用 DISTINCT 应该是一条基本规则，原因在于，DISTINCT 可能会掩盖我们“遗漏了连接条件”的错误。发现重复数据容易，但发现数据不准确很难。

上面的例子中，DISTINCT 不仅会剔除同一个客户的多次购买，也有可能剔除掉名字相同的不同客户的购买，所以应该同时返回具有唯一性的 customer_id 和 customer_name，以保证得到蝙蝠车买家的完整清单。

避免 DISTINCT 的方法：

从逻辑上讲，JOIN 两个或多个表建立了新的关系。最初的查询方案中，（逻辑上）新生成的关系在 customer_name 上并没有唯一性约束，因此查询结果中可能出现同一客户的重复记录。

使用 DISTINCT 的目的是使每个客户在结果集中只出现一次，而我们知道每个客户在 customers 表中只会出现一次，因此我们可以设定某些过滤条件从单一 customers 表中检索记录（关键的区别）。

“购买了 Batmobile”是一个存在性的条件，可以用存在性测试

使用无关（非关联）子查询

新版的查询中执行了隐式的 DISTINCT（IN 会隐式地剔除重复项）

注意：每当对大量记录做存在性检查时，选择 IN 还是 EXISTS 需要斟酌

  • 如果 Gotham 市的客户很少，那么先通过 city = 'Gotham' 筛选就能够取得比较好的效果
  • 如果 Gotham 市的客户很多，但是只有很少一部分客户买了 Batmobile，因此先筛选出购买了 Batmobile 的客户效果就会比较好

2.3    大数据量查询

越快剔除不需要的数据，查询的后续阶段必须处理的数据量就越少，自然查询的效率就越高

为了提高查询的效率，可以仅对代码中非共用的表使用 UNION，然后配合筛选条件，把 UNION 语句降级为内嵌视图

引入一个新的 Schema order_status(order_id, status_date, status)

问题：找到所有尚未标记为完成状态的订单（假设所有交易都已终止）的订单号、客户名、订单的最后状态，以及设置状态的时间

我们想让 orders 和 order_status 的 order_id 相等，优化器不一定能够感知这一点，它可能依然先执行子查询，但在使用子查询的结果之前，还必须将 orders 和 order_status 两个表连接起来

即使最基本的查询完全相同，优化器在返回大量数据时所选择的路径仍可能与返回少量数据时完全不同

总结：尽早过滤掉不需要的数据

总结：当查询的结果集很大时，索引未必必要
