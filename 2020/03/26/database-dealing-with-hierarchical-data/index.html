数据库 | 存储层次结构数据

发表于 2020-03-26 更新于 2020-03-27 分类于 数据库 (Database)
如题

层次结构之所以复杂，主要原因不是“组件之间关系的表达”，而是“访问树（walk a tree）的方式”。我们通常访问树的全部或部分节点，并以特定顺序返回这些节点。访问树通常由 DBMS 引擎以过程性的方式实现，而过程性的操作正是违背关系理论的主要表现之一。

1    树状结构 vs. 主从关系

所谓“主 / 从关系”就比如订单和订单明细

  • 树状结构的保存只需要一个表 —— 因为所有节点的类型完全相同（暂时不考虑叶节点），换句话说，表与它本身之间有种主 / 从关系
  • 深度：层次结构中，与根节点的距离是重要的信息
  • 所有权：主 / 从关系中，可以有明确的外键完整性约束。这一点可能会导致树状结构中森林的产生，即在同一个表中描述多个独立的树
  • 多重父节点

树中有两种实体类型：节点（可能包含信息量更大的叶节点的子类型）和节点之间的连接

层次结构的实际案例

  • 金融机构的风险敞口（risk exposure）：金融机构投资基金，基金又会投资其他的基金 …
  • 档案位置：建筑 / 楼层 / 房间 / 柜子 / 架子 / 箱子 / 档案夹 / 档案
  • 原料使用

树结构要解决的几种查询

  • 自顶向下
  • 自底向上
  • 聚合

总结：只要对象的类型相同、而对象之间的层数可变，其关系就应该被建模为树结构

如果树的深度固定 / 所有叶节点的深度相同，只需要多次连接即可，问题就在于这两个条件很难满足，多数情况下需要处理的是深度可变的情况。

2    用 SQL 数据库描述树结构

  • 邻接模型 (adjacency model)：父记录的 ID 作为子记录的属性（想想并查集 (Union-Find)）
    • Schema：adjacency_model(id, parent_id, description, commander)
    • 索引
      • 在 ID（主键）上的唯一索引
      • 在 PARENT_ID 上的索引
      • 在 COMMANDER 上的索引
  • 物化路径模型（路径枚举模型）(materialized path model)：在每个节点的记录中附加上它在树中位置的描述数据。在物化路径模型中，某个节点的路径是其所有后代的前缀，属于同一祖先的节点有非空的公共前缀
    • Schema：materialized_path_model(materialized_path, description, commander)
    • 索引
      • 在 MATERIALIZED_PATH（主键）上的唯一索引
      • 在 COMMANDER 上的索引
  • 嵌套集合模型 (nested sets model)：每个节点被赋予一对数，某节点的两个数所圈定的区间能够包括所有后代节点的区间（赋予 left_num 和 right_num 的方式就是 DFS 中计算活动区间的方式），叶子元素是 left_num + 1 == right_num 的元素
    • Schema：nested_sets_model(description, commander, left_num, right_num)
    • 复合主键：(left_num, right_num)，在 commander 上的索引
  • 嵌套间隔模型 (nested interval model)：这里不讨论

3.3.2    散布在各层的百分比

假设我们经营魔药。每种魔药由多种成分 (ingredient) 组成，处方 (recipe) 列出成分及百分比。处方可以共享某种“基础魔药”，以复合成分 (compound ingredient) 的形式表示

如何建立层次模型？物化路径模型无法使用，因为与在军队层次中具有单一、明确定义的位置的部队不同，每种成分可以用在多种魔药中。“扁平化”地保存魔药，并用另一个表保存魔药与各种基础成分的关联也不行，因为存在着共享的“基础魔药”，如果“基础魔药”的处方改变，会影响所有使用它的魔药。

当然，实际上只有两个表：

  • components(component_id, component_name, component_type)
    • component_type 用于指示 component 是基本成分 (I) 还是复合成分 / 处方 (R)
  • composition(component_id, recipe_id, pct)
    • composition 是一个药方中的一个条目，因此要使用 recipe_id 指示所属的药方，另外还需要 component_id 指示这个条目使用了那种成分，及其百分比
    • recipe_id 和 component_id 联合起来作为 composition 的主键
