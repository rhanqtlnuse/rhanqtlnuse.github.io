【译】To Be Explicit

发表于 2020-04-05 更新于 2020-04-06
关于代码构造的一篇文章，原作者 Martin Fowler

原作者：Martin Fowler

原文链接

对于构造某件事物而言，软件是一个奇怪的媒介。因为几乎没有什么物理因素使你按照某种方式设计，许多设计决策违背了任何形式的客观分析。设计的重要性并不在于软件如何运行，而在于它有多容易变更 (change)。当软件如何运行重要时，易变更性 (ease of change) 会是保证良好性能的最大因素。

来自可变更性 (changeability) 的驱动力是为什么设计清楚地展示程序做了什么 —— 以及如何做 —— 如此重要的原因。毕竟，在不了解一件事物做了什么的情况下，修改它是很困难的。由此产生的一个有趣的必然结果是，人们通常使用特定的设计，因为这些设计容易变更，但是当他们使得程序难以理解时，效果却会与期望相反。

使用字典相当有吸引力，因为它让你在不变更 Person 类定义的情况下变更存入某个 Person 对象的数据。如果你想增加一个电话号码，可以在不更改原始代码的情况下做到。

然而，字典并没有使得修改代码更加容易。如果我想要使用 Person 对象，我无法明确其中都有什么。我必须查看整个系统的代码才能知道某人保存了家属 (dependent) 的数量。如果家属的数量在类中显式地声明，我只需看 Person 类的代码就能够了解它都支持什么操作。

关键的原则是明确 (explicit) 的代码更加容易理解 —— 也使得修改代码更加容易。就像 Kent Beck 所说，明确的代码能够表明作者的意图（译者注：原文是“the explicit code is intention revealing”）。

这个字典示例规模很小，但是其中的原则对于几乎任何规模的软件设计都成立。

事件 (Event) 和明确的调用 (Call)

下面有另一个规模稍微大一点的例子。为了在不同的模块 (module) 之间通信 (communicate)，许多平台都支持“事件”这样的概念。假设我们有一个预定 (reservation) 模块，当预定被取消时，需要获得 person 模块，并向那个人发送一封电子邮件。

我们可以定义影响一个预定的有趣的事件，任何在一个事件发生时想要做任何事情的对象都能够创建一个处理器 (handler) 对事件做出响应。这个办法很有吸引力，因为你不需要修改 Reservation 类来在取消一个预定时让某些其他的事情发生。只要其他对象设置了这个事件的处理器，就可以很容易的在这些时候扩展行为。

然而，使用事件是有代价的 —— 仅仅通过阅读 Cancel 方法的代码，我无法了解在取消预定时发生了什么。为了查明到底发生了什么，我必须得搜索所有对这个事件有 handler 的代码。明确的代码清楚地显示了 Cancel 方法中取消预订的影响，代价就是，当我需要变更行为时，必须修改 Reservation 类的代码。

我见过一些大量使用时间的代码，问题就是很难确定当你调用方法时程序做了什么。而在调试的时候，这会变得非常难受 —— 行为会突然出现在你意想不到的地方。

我并不是说你不应该使用事件。事件让你在不变更类的前提下执行行为，在使用无法修改的库类时这会很有用。另外，使用事件的方式不会在事件的触发者和事件的响应者之间建立依赖关系，这使得这种方式很有价值。当两个类在不同的包，并且你不想增加依赖时，这中没有依赖的方式很有用。经典的案例是，当你想要当你想要修改表现层中一个窗口的时候。事件让你在保留领域对象和表现层的至关重要的分离的情况下实现。

这些因素都暗示应该使用事件，但是当这些都不成立的时候，明确性的缺乏就成了更主要的。因此，在两个应用类 (application class) 能够了解彼此的时候，我不愿意使用事件在他们之间进行通信。

如你所见，明确性并不总是影响设计决策的最主要因素。在这个例子中，包和依赖因素也都很重要。人们经常低估明确性的价值。有时我会增加依赖让代码更加明确，但是每种情况都有自己的权衡点 —— 设计一贯如此。

在设计中，明确性并不是绝对要求，但是聪明的设计常常因不够明确而变得难以使用。在某些情况下，付出的代价是值得的，但总要考虑一下。在过去的几年中，我倾向于选择更明确的设计，因为我对好的设计的看法已经发展（希望是朝着正确的方向）。

译者注

总结一下这篇文章的观点。

明确的代码 不明确的代码
优点 更容易理解
效率可能更高		 灵活
缺点 不够灵活 难以理解
效率可能更差

另外，使用字典表示对象并且键为字符串时，可能会因输错字符串而产生一些意想不到的错误（毕竟不是所有的语言都像 Ruby / Lisp 一样有“符号”这样的数据类型（定义字符串常量似乎也不太现实，）。不过，完备的测试能够解决这个问题）。

最终还是要根据具体的需求选择合适的方法。
