常见的命名方式

发表于 2020-04-28 更新于 2020-04-29
如题

1    驼峰式命名法（Camel Case）

“Camel Case”一词来自 Perl 语言中普遍使用的大小写混合格式，而 Larry Wall 等人所著的畅销书《Programming Perl》（O’Reilly 出版）的封面图片正是一匹骆驼。

驼峰式命名法是一种惯例，为的是增加可读性，并无强制要求。一旦选用或设置好命名规则，在程序编写时应保持一致格式。

驼峰式命名法的基本规则是：单词之间不以空格断开（例如，camel case），也没有其他标点符号。包括小驼峰式命名法（lower camel case）和大驼峰式命名法（upper camel case）两种。

  • 小驼峰式命名法是指第一个单词全小写，其余的单词首字母大写，这种命名方式也称为“Dromedary case”。比如，firstName；
  • 大驼峰式命名法是指所有单词的首字母都大写，这种命名方式也被称为“Pascal 命名法”。比如，ActiveRecord

2    蛇形命名法（Snake Case）

蛇形命名法的使用可以追溯到上世纪 60 年代，比较著名的就是《C 程序设计语言》中，但是一直没有正式的名字。在 Usenet 中，“snake_case”这个名字首次被看见是在 2004 年的 Ruby 社区，使用者是 Gavin Kistner，他写道：

“BTW… what do you call that naming style? snake_case? That’s what I’ll call it until someone corrects me.”

但是，前英特尔工程师杰克·达尔格伦（Jack Dahlgren）在 Quora 上表示，他在 2002 年在英特尔内部使用过该术语（可能与微软工程师进行了对话）。

在蛇形命名法中，单词之间用下划线 _ 隔开，单词的大小写取决于具体的场景。比如，在 Java 中，静态常量和枚举值是全大写的，单词之间用 _ 隔开；在 Ruby 中，用全小写表示变量和方法名；还有些情况下是第一个单词的首字母大写，其余全小写，比如 Wikipedia 的词条命名。

3    Kebab 命名法（Kebab Case）

Kebab 命名法的起源没查到。

Kebab 命名法与蛇形命名法很相似，不同之处在于 Kebab 命名法使用连字符 - 来分隔单词。如果单词全都是大写的，也可以称为“火车命名法”（TRAIN-CASE）。

4    匈牙利命名法（Hungarian Notation）

由 Charles Simonyi 发明，此人后来成了微软的总设计师（我猜这就是为什么微软的 API 和文档中会使用匈牙利命名法），更详细的信息参见 [4]。

首次在 BCPL 语言中被大量使用。由于 BCPL 只有机器字这一种数据类型，因此这种语言本身无法帮助程序员来记住变量的类型。匈牙利命名法通过明确每个变量的数据类型来解决这个问题。

在匈牙利命名法中，一个变量名由一个或多个小写字母开始，这些字母有助于记忆变量的类型和用途，紧跟着的就是程序设计师选择的任何名称。这个后半部分的首字母可以大写，以区别前面的类型指示字母。

匈牙利命名法分为系统匈牙利命名法 与应用匈牙利命名法，二者的区别在于前缀的目的。

  • 系统匈牙利命名法的前缀代表变量的实际数据类型。例如，lAccountNum 的前缀 l 表示这个变量是一个长整数（l）
  • 应用匈牙利命名法的前缀代表变量的目的。例如，strName 的前缀 str 表示这个变量是一个字符串，但是没有指明这个字符串是如何实现的。

Simonyi 建议的大多数前缀都是自然语义的。下面几个是来自原始论文的：

  • p 表示这个变量是一个指针，语义信息很少；
  • d 表示两个值的区别，例如，dY 可能代表一个图形沿 Y 轴的距离，而一个仅仅叫做 y 的变量可能是一个绝对坐标。这完全是自然语义的；
  • sz 表示一个以 null 或 \0 结束的字符串。在 C 中，这包含一些语义信息，因为 C 语言中 char * 类型的变量可能是一个指向单个字符的指针，也可能是一个字符数组，或是一个 \0 结束字符串。
  • w 标记一个变量是一个字。这基本上没有包含什么语义信息，因此可以被看作一种系统匈牙利命名法。
  • b 标记了一个字节，和 w 对比可能有一些语义信息，因为 C 语言中，只有 char 型（以及 signed/unsigned char）是一个字节长的，这些类型有时候被用来保存数值而非字符。这个前缀可以明确某个变量保存的是字符还是数值。
    • 缺点是，可能与 bool 混淆

这种命名法通常使用小写字母开头用来助记，但是并没有对助记符本身作规定，只要它们在代码主体中保持一致就可以了。有几种被广泛使用的习惯。

5    结束语

使用哪种命名法完全取决于自己的喜好或是代码风格规定，通常情况下编程语言会有一些关于命名的建议，但是一般不会有强制的要求（例外是 Ruby 中要求类名和模块名的首字母必须大写，Haskell 中要求变量名和函数名的首字母必须小写、类型名和类型类名的首字母必须大写），只要在整个项目中能够保持一致就好。

还有一些其他的有趣的命名方式。比如我个人很喜欢的：Lisp 和 Ruby 中，可以在函数 / 方法名的后面加上 ! 表示这是一个有危险（或副作用）的方法，加上 ? 表示这是一个测试，个人认为这种方式更加直观。另外，在 Haskell 中，可以在方法名的后面加上一个 ' 表示这个函数与原来的函数名所代表的函数有略微不同的行为。
