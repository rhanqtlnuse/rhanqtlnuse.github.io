面向对象设计基本概念：耦合和内聚

满足需求的情况下模块之间尽可能地相互独立

1 耦合 (Coupling)

1.0 模块通信：连接

连接的强度取决于数量、调用次数、参数数量、方向等

两个对象之间的连接有如下几种可能：

  • 方法调用
  • 访问属性
  • 共享数据
  • 对象 A 持有对象 B 引用
  • 继承
  • 隐式

耦合分析的标准

  • 复杂度（参与者、范围）

1.1 什么是耦合？

耦合描述的是两个模块之间关系的复杂程度。

Coupling is the manner and degree of interdependence between software modules; a measure of how closely connected two routines or modules are; the strength of the relationships between modules. [1] Coupling is the measure of the degree of interdependence between the modules. A good software will have low coupling. [2]

1.2 耦合的分类

耦合分为 6 种，分别是：数据耦合、印记耦合、控制耦合、重复耦合（有一些资料没有这一项，取而代之的是“外部耦合”，这里以南京大学《软件工程与计算》（卷二）为准）、公共耦合、内容耦合，其中数据耦合是最好的（耦合性最低），内容耦合是最差的（耦合性最高）。

1.2.1 数据耦合 (Data Coupling)

1.2.1.1 定义

If the dependency between the modules is based on the fact that they communicate by passing only data, then the modules are said to be data coupled. In data coupling, the components are independent to each other and communicating through data. Module communications don’t contain tramp data. This data item should be problem-related and not used for the control purpose.

模块之间仅通过传递数据进行通信，并且传递的数据仅包含必要的数据，也不用于控制。

1.2.1.2 示例

1.2.2 印记耦合 (Stamp Coupling)

1.2.2.1 定义

In stamp coupling, the complete data structure is passed from one module to another module. Therefore, it involves tramp data. It may be necessary due to efficiency factors - this choice made by the insightful designer, not a lazy programmer.

模块之间通过传递完整的数据结构进行通信，而被调用的模块只在该数据结构的个别组件上进行操作。

反过来，如果两个方法之间建立了印记耦合，那么它们之间肯定传递了复杂数据。

常见的复杂数据有集合（数组、链表等）和复合数据类型（结构体、类等）

1.2.3 控制耦合 (Control Coupling)

1.2.3.1 定义

If the modules communicate by passing control information, then they are said to be control coupled. It can be bad if parameters indicate completely different behavior and good if parameters allow factoring and reuse of functionality.

模块之间通过传递控制信息进行通信，并且控制信号会影响被调用者执行路径分支的选择。（如果控制信息有多余的信息呢？）

判断控制信号的标准：

1.2.5 公共耦合 (Common Coupling)

1.2.5.1 定义

The modules have shared data such as global data structures. The changes in global data mean tracing back to all modules which access that data to evaluate the effect of the change. So it has got disadvantages like difficulty in reusing modules, reduced ability to control data accesses and reduced maintainability.

模块之间有共享的数据（如全局数据结构）。

1.2.5.2 解释

公共耦合的关键是公共环境。

比如，类 A 和类 B 都访问了类 C，其中类 C 是 Java Bean，只有私有成员和针对每一个变量的 getter 和 setter，那么 A 和 B 之间就不是公共耦合。（那是什么耦合？）

1.2.6 内容耦合 (Content Coupling)

1.2.6.1 定义

In a content coupling, one module can modify the data of another module or control flow is passed from one module to the other module.

一个模块直接修改或者依赖于另一个模块的内容。

1.2.6.2 示例

模块 A 利用 goto 语句跳转到模块 B 的内部；模块 A 更改模块 B 的内部私有数据；模块 A 能够直接更改模块 B 的代码（COBOL 语言就允许这样做）

GOTO Statement considered harmful

破坏了模块的独立性

1.2.7 总结

只要软件模块之间有依赖关系，就会不可避免地产生耦合。但其中，数据耦合是最理想的，印记耦合和控制耦合是可接受的，公共耦合应该控制使用.重复耦合、内容耦合应该避免使用。

2 内聚 (Cohesion)

2.1 什么是内聚？

内聚表达的是一个模块内部的联系的紧密性。

In computer programming, cohesion refers to the degree to which the elements of a module belong together. Thus, cohesion measures the strength of relationship between pieces of functionality within a given module. For example, in highly cohesive systems functionality is strongly related. [3]

内聚服从于耦合，高内聚是为了实现低耦合。

2.2 内聚的分类

内聚性越高越好，越低越不易实现变更和复用。

2.2.1 信息内聚 (Information Cohesion)

2.2.1.1 定义

模块内有一系列操作，每个操作都有各自的入口点和出口点，每个操作的代码相对独立，而且所有的操作都是在相同的数据结构上完成，但是却形成一个抽象的整体。

2.2.1.2 示例

信息内聚的模块主要用来实现抽象的数据类型。

2.2.2 功能内聚 (Functional Cohesion)

2.2.2.1 定义

Functional cohesion is said to exist if different elements of a module cooperate to achieve a single function. It is an ideal situation.

模块只执行一个单一目的的操作。

要求一个模块只能有一个目标。

目标一定跟需求相对应

2.2.2.2 解释

Suppose a module shows functional cohesion and we are asked to …, then we would be able to tell it using a single sentence.

2.2.2.3 示例

计算平方根、计算最短路径、压缩数据库

a module containing all the functions required to manage employees’ pay-roll exhibits functional cohesion.

2.2.3 通信内聚 (Communicational Cohesion)

2.2.3.1 定义

A module is said to have communicational cohesion, if all functions of the module refer to or update the same data structure.

模块内的操作都是针对同一数据进行的。

2.2.3.2 示例

update record int the database and send it to the printer.

查询书名、查询价格、查询出版社和查询作者。

2.2.4 过程内聚 (Procedural Cohesion)

2.2.4.1 定义

A module is said to have procedural cohesion, if the set of functions of the module are all part of a procedure (algorithm) in which certain sequence of steps have to be carried out for achieving an objective.

Elements of procedural cohesion ensure the order of execution. Actions are still weakly connected and unlikely to be reusable.

模块执行的多个操作，是解决同一个问题的不同步骤，这些子过程之间是有顺序关系的，即，前一个子过程的输出是下一个子过程的输入。

所有的代码都有一个相同的问题。

2.2.4.2 示例

the algorithm for decoding a message.

calculate student GPA, print student record, calculate cumulative GPA, print cumulative GPA.

保养车的过程依次为洗车、消除凹痕、打磨、上蜡

2.2.5 时间内聚 (Temporal Cohesion)

2.2.5.1 定义

A module is said to have temporal cohesion, if it contains functions that are related by the fact that all the functions must be executed in the same time span. The elements are related by their timing involved.

模块中执行一系列操作，但是这些操作在同一时间段内发生。

2.2.5.2 示例

This cohesion contains the code for initializing all the parts of the system. Lots of different activities occur, all at init time.

2.2.6 逻辑内聚 (Logical Cohesion)

2.2.6.1 定义

模块执行一系列逻辑上相似但没有直接关联的操作。

2.2.6.2 示例

A component reads inputs from tape, disk, and network. All the code for these functions is in the same component. Operations are related, but the functions are significantly different.

a set of print functions generating different output reports are arranged into a single module.

典型的是通过 switch-case 进行路径选择（控制耦合？）

2.2.7 偶然内聚 (Coincidental Cohesion)

2.2.7.1 定义

A module is said to have coincidental cohesion if it performs a set of function or tasks that relate to each other very loosely. In this case, the module contains a random collection of functions. The elements have no conceptual relationship other than location in source code. It is likely that the functions have been put in the module out of pure coincidence without any design or thought. It is accidental and the worst form of cohesion.

模块中执行的操作毫无关系，只是恰好堆砌在这个模块内。

2.2.7.2 示例

print next line and reverse the characters of a string in a single component.

2.2.9 总结

功能内聚和信息内聚是最好的两种，而且这两者的出发点不同，一个完全以功能（行为）为依据进行模块分解，一个以数据与功能间的相互支撑为依据进行模块分解，不可相互比较。一般而言，函数与过程应该是功能内聚的（信息内聚不适用于函数与过程），模块应该是信息内聚或功能内聚的，面向对象方法中的类应该是信息内聚与功能内聚兼顾的。

偶然内聚和逻辑内聚是不能接受的，如果软件中出现这两类内聚，一定要进行优化。

通信内聚、过程内聚、时间内聚这三种类型的内聚也是各自具有不同的出发点（一个是相同的数据，一个是相同的问题，一个是相同的时间），无法相互比较，这三种类型的内聚都是可以接受并且不可避免的。

但是因为通信内聚、过程内聚和时间内聚比功能内聚、信息内聚要差一些，所以在设计时还需要限定这三种内聚类型的使用。

中间三个是等价的

用一句话描述

  • 如果刚好一个动词 — 功能内聚
  • 超过一个动词 — 多于一个功能，可能是通信内聚或者顺序内聚
  • 如果出现首先、其次，可能就是顺序内聚/时间内聚
  • 如果发现多个主体，可能是逻辑内聚
  • 初始化、清除等，可能是时间内聚

3 耦合和内聚的区别

耦合 内聚
Coupling is also called Inter-Module Binding. Cohesion is also called Intra-Module Binding.
Coupling shows the relationships between modules. Cohesion shows the relationship within the module.
Coupling shows the relative independence between the modules. Cohesion shows the module’s relative functional strength.
While creating, you should aim for low coupling, i.e., dependency among modules should be less. While creating you should aim for high cohesion, i.e., a cohesive component/module focuses on a single function (i.e., single-mindedness) with little interaction with other modules of the system.
In coupling, modules are linked to the other modules. In cohesion, the module focuses on a single thing.

5 如何实现高内聚、低耦合？

按照功能分解的方式进行模块分割能够实现高内聚的软件设计，但并不一定能降低耦合，高内聚要从属于低耦合，低耦合才是设计的第一准则，所以按照功能分解并不是最佳的方式。

  • global variables considered harmful
  • to be explicit（Martin Fowler — To Be Explicit）

显式 vs. 可修改性

  • do not repeat（打印账单的例子）
  • Programming to Interface (Stevens — Structured Design)
  • Demeter 法则
  • Interface segregation principle
  • Liskov
